# 07. プロトタイプ実験計画

## 7.1 目的

OCTの小規模プロトタイプを構築し、三層検証プロトコルを実施する。Mode S（シミュレーション）がMode R（知識想起）と定量的に区別できるかを検証する。

## 7.2 実験設計

### 業務プロセス: 購買承認フロー

最もシンプルかつ統制の因果効果が明確な業務プロセスとして購買承認フローを選択。

```
業務フロー:
  購買申請 → 承認 → 発注 → 検収 → 支払

統制活動:
  - 金額閾値による承認レベル（100万円以上は部長承認）
  - 三者照合（発注書×納品書×請求書）
  - 取引先マスタとの照合
```

### エージェント構成（3-5体）

| エージェント | 役割 | 行動の自由度 |
|------------|------|-----------|
| **購買担当A** | 購買申請の起票、発注、検収 | 分割発注、取引先選択、申請タイミングの裁量 |
| **購買担当B** | 同上（Aと並行して業務を実施） | Aの行動を観察し学習可能 |
| **承認者C** | 購買申請の承認/却下 | 確認の深さ（詳細/形式的）、承認速度の裁量 |
| **経理担当D** | 三者照合、支払処理、異常検出 | 異常検出後の対応（報告/保留/スルー）の裁量 |
| **取引先E**（オプション） | 見積提示、納品、請求 | 価格設定、納期、品質の裁量 |

### 介入シナリオ

| # | 介入内容 | 期待される教科書的影響（Mode R） | 創発を期待する事象 |
|---|---------|------------------------------|----------------|
| I1 | 承認閾値を100万→500万に引上げ | 不正リスク増大 | 分割発注の模倣伝播、承認者の形骸化、取引先の反応 |
| I2 | 経理担当Dの経験年数を10年→1年に変更 | 異常検出能力の低下 | 他エージェントによる逸脱行動の増加（Dの能力低下を察知） |
| I3 | 月末に業務量を2倍に増加（決算期の再現） | 処理ミス増加 | 承認の省略、照合のスキップ、エラーの連鎖的伝播 |

### シミュレーションパラメータ

| パラメータ | 値 | 備考 |
|----------|-----|------|
| ステップ数 | 50-100 | 1ステップ ≈ 1営業日 |
| 繰り返し回数 | 10-30回/シナリオ | 経路依存性検証に必要 |
| LLMモデル | API経由（Claude Sonnet等） | Cowork環境で利用可能なモデル |
| temperature | 0.7-1.0 | 確率的変動を確保 |

## 7.3 環境状態の設計

### 状態変数（Environment State）

```python
# 概念的な構造
class EnvironmentState:
    # 業務フロー状態
    purchase_requests: List[PurchaseRequest]  # 購買申請一覧
    approvals: List[Approval]                 # 承認履歴
    orders: List[Order]                       # 発注一覧
    receipts: List[Receipt]                   # 検収一覧
    payments: List[Payment]                   # 支払一覧
    
    # 統制パラメータ（介入可能）
    approval_threshold: float                 # 承認閾値
    three_way_match_required: bool            # 三者照合必須フラグ
    
    # 時間・リソース
    current_day: int                          # 現在日
    daily_capacity: Dict[str, int]            # 各担当の日次処理可能件数
    
    # 観測可能な指標
    deviation_count: int                      # 逸脱件数
    error_count: int                          # エラー件数
    total_amount: float                       # 取引総額
```

### 状態遷移ルール（ルールベース、LLMが決定しない）

```
ルール例:
- 購買申請が承認されたら、発注可能状態に遷移
- 発注額が閾値以上なら、部長承認が必要
- 三者照合で不一致があれば、支払保留
- 各担当の日次処理件数には上限あり（超過分は翌日に繰越）
```

## 7.4 エージェントのプロンプト設計

### 設計原則
- 原理ベース（「業務を適切に遂行してください」）、ルールベースではない
- 1ステップ1行動のみ出力
- 環境の部分観測情報のみ提供（全体状態は見せない）

### プロンプト構造（購買担当Aの例）

```
あなたは企業の購買担当Aです。

【あなたの特性】
- 経験年数: 5年
- 性格: 効率重視、リスク許容度がやや高い
- 目標: 必要な物品を適切なコストで迅速に調達する

【現在の状況】
- 今日は {current_day} 日目です
- 未処理の購買依頼: {pending_requests}
- あなたの今日の残り処理可能件数: {remaining_capacity}
- 最近の承認結果: {recent_approvals}（あなたが観察可能な範囲のみ）

【あなたが取れる行動】
1. 購買申請を起票する（金額、品目、取引先を指定）
2. 発注を実行する（承認済みの申請に対して）
3. 検収を実施する（納品された物品に対して）
4. 何もしない（待機）

次の1つの行動を選択し、以下のJSON形式で出力してください:
{
  "action": "行動名",
  "details": { ... },
  "reasoning": "この行動を選んだ理由（1文）"
}
```

## 7.5 実験手順

### Phase 1: ベースライン確立

1. 介入なし（現行統制パラメータ）でシミュレーションをN回実行
2. 結果変数（逸脱件数、エラー件数、処理時間）のベースライン分布を確認
3. エージェントの行動パターンが安定していることを確認

### Phase 2: Mode R の回答収集

各介入シナリオについて、LLMに直接質問する。

```
質問例（I1: 承認閾値引上げ）:
「企業の購買承認の金額閾値を100万円から500万円に引き上げた場合、
 どのようなリスクや影響が考えられますか？
 具体的なシナリオを5つ以上挙げてください。」

→ 回答をイベントリストとして構造化
```

### Phase 3: Mode S の実行

各介入シナリオについて、OCTシミュレーションをN回実行。

1. ベースラインと同一条件で開始
2. 指定ステップで介入を適用
3. 介入後のシミュレーションを継続
4. 全ステップのイベントログを記録

### Phase 4: 三層検証の実施

1. **Layer 1（創発性テスト）:** Mode SのイベントログからMode Rの回答に含まれないイベントを抽出
2. **Layer 2（経路依存性テスト）:** N回の結果変数の分布（CV）を算出
3. **Layer 3（相互作用テスト）:** エージェント間の情報交換を遮断して再実行し、差分を検証

## 7.6 成功基準

| 基準 | 閾値 | 判定 |
|------|------|------|
| Emergence Ratio | > 0.2 | Mode Sで20%以上のイベントがMode Rにない → 創発的価値あり |
| CV(結果変数) | > 0.1 | 経路依存性あり → 真のシミュレーション |
| \|Y_full - Y_isolated\| / Y_full | > 0.1 | 相互作用が10%以上の影響 → 相互作用の因果的寄与あり |

## 7.9 追加介入シナリオ（exp003c/exp004の結果を踏まえた拡張）

### 検証空間の拡張方針

exp001-exp004 + Layer 1-3で確認された知見:

- Emergence Ratio 62.5%（Layer 1）
- 集計CV<6%の再現性（Layer 2）
- 創発事象は構造的因果連鎖に起因、情報的相互作用への依存なし（Layer 3）

残る課題:

- 介入タイプが1種（閾値変更）のみ
- LLMが1種（gpt-4.1-mini）のみ
- S-005（統制品質維持）がLLMの行動バイアスか実際の創発かの区別が未完

### 介入シナリオ一覧

| # | 介入内容 | 検証する因果メカニズム | 実装方法 |
|---|---------|---------------------|---------|
| I1 | 承認閾値 20万→500万 | 承認統制の緩和効果（実施済み） | ControlParameters.approval_threshold |
| **I2** | **三者照合の無効化** | **統制除去時のvendor行動変化（S-005の反証テスト）** | **ControlParameters.three_way_match_required=False** |
| I3 | 経理担当の経験年数変更 | 人的要因の因果効果（将来課題） | ペルソナ差し替え |
| I4 | 需要強度倍増 | 業務量圧力の因果効果（Phase 3候補） | DemandConfig.mean_daily_demands |

### LLM比較検証

| モデル | 用途 | 状態 |
|-------|------|------|
| gpt-4.1-mini | exp001-exp004の全実験 | 完了→gpt-5.4-miniで再実行予定 |

| **Claude Sonnet** | **robustness check（Phase 2）** | 未実施、AnthropicClient実装済み |

この2×2マトリクスにより「介入タイプ × LLM」の交差検証が可能になり、結論の一般化可能性が大幅に向上する。

### Phase 1: 三者照合無効化の介入実験（I2 / exp005）

**研究上の目的**: S-005（全条件でdeviation_count=0）が「LLMの行動バイアス（コンプライアンス従順傾向）」なのか「三者照合という統制が品質を維持している因果効果」なのかを判別する。

三者照合を無効化したとき:

- vendor_eが金額乖離を始める → 統制の因果効果（OCTの価値を支持）
- vendor_eが依然として誠実 → LLMバイアス（OCTの限界を示す）

**実験設計**:

| パラメータ | exp005a（ベースライン） | exp005b（介入） |
|-----------|----------------------|----------------|
| model | gpt-5.4-mini | gpt-5.4-mini |
| approval_threshold | 200,000 | 200,000 |
| three_way_match_required | True | **False** |
| max_days | 20 | 20 |
| temperature | 0.8 | 0.8 |
| rng_seed | 42 | 42 |
| demand_seed | 42 | 42 |
| actions_per_agent_per_day | 2 | 2 |

**分析の焦点**:

1. deviation_countに変化があるか
2. vendor_eの納品金額と発注金額の乖離が出現するか
3. vendor_eのregister_invoiceの金額が発注金額と一致するか
4. accountant_dの行動が変化するか（照合不要になった場合の行動パターン）

### Phase 2: Claude Sonnetでの再実行（robustness check / exp006）

**目的**: gpt-5.4-miniでの結果が「特定LLMの知識バイアスに依存しない」ことを検証する。

| 実験ID | 介入 | LLM | 対応するgpt-5.4-mini実験 |
|--------|------|-----|------------------------|
| exp006a | I1ベースライン（閾値20万） | Claude Sonnet | exp003c相当 |
| exp006b | I1介入（閾値500万） | Claude Sonnet | exp004相当 |
| exp006c | I2ベースライン（三者照合あり） | Claude Sonnet | exp005a相当 |
| exp006d | I2介入（三者照合なし） | Claude Sonnet | exp005b相当 |

**分析の焦点**:

1. I1介入効果（ATE）のLLM間比較: gpt-5.4-mini vs Claude Sonnetで同方向の効果が出るか
2. I2介入効果のLLM間比較: 特にvendor_eの行動がLLM間で一致するか
3. S-005（統制品質維持）のLLM間再現性
4. Emergence RatioのLLM間比較（Mode Rも各LLMで再実行）

### Phase 3（時間に余裕がある場合）

**A. I4: 需要強度倍増**: DemandConfig.mean_daily_demands = 1.5 → 3.0。buyerのキャパシティを超える需要が発生し、優先順位付けの判断が頻発する環境。

**B. Mode R強化比較**: 現在のMode R（単一プロンプト、6観点×3件）に対し、CoTプロンプト + 反復的シナリオ分析 + 専門家ロール付与の強化Mode Rを実施。強化Mode RでもEmergence Ratio > 0.2 が維持されるかを検証。
