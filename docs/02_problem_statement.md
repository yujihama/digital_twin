# 02. 課題認識と研究動機

## 2.1 従来の因果推論の構造的限界

### DAGベース因果推論の前提と制約

従来の因果推論（Pearl SCM / Rubin 潜在結果フレームワーク）は以下を要求する。

1. **変数の事前定義:** 処置変数T、結果変数Y、交絡変数Wを研究者が明示的に定義する
2. **因果グラフ（DAG）の構築:** 変数間の因果関係を有向非巡回グラフとして指定する
3. **因果的十分性（Causal Sufficiency）:** 全ての共通原因が観測されている仮定
4. **識別条件の検証:** 無交絡仮定（ignorability）、重複仮定（positivity）等

### これらが内部統制の文脈で問題となる理由

| 前提 | 内部統制での問題 |
|------|----------------|
| 変数の事前定義 | 「組織文化」「暗黙知」「時間圧力」「担当者間の信頼関係」は連続変数や二値変数として定式化すること自体が困難 |
| DAGの構築 | 統制の因果パスは複雑。承認統制→担当者の行動変化→取引先の反応→経理の検出→報告の抑制/エスカレーション、のような多段階の連鎖は事前に網羅的に定義できない |
| 因果的十分性 | 未観測の交絡（個人のモチベーション、部門間の政治力学等）が常に存在 |
| 識別条件 | 統制の配置はリスクプロファイルに依存（内生性）。RCTは倫理的・実務的に不可能 |

### この課題は因果推論コミュニティでも認識されている

- **高次元問題:** 変数数が増えるとDAG空間の探索が指数的に困難（Causal-StoNet, ICLR 2024）
- **定式化困難な因果関係:** 複雑システムでは事前のDAG定義が根本的にボトルネック
- **FCI等の拡張:** 未観測交絡に対処するが、出力はPAG（部分的祖先グラフ）であり一意な因果構造を特定できない

## 2.2 LLMエージェント研究の現在の焦点と見落とし

### 現在の焦点: 人間の業務代行

LLMエージェント研究の大半は「人間が行うタスクをLLMが代行する」ことに焦点がある。

- コード生成、文書作成、情報検索
- 監査判断、医療診断支援、法的分析
- マルチエージェント協調（MetaGPT, ChatDev等）

### 見落とされている可能性: 環境のシミュレーション

LLMの能力を「タスク実行」ではなく「環境のシミュレーション」に使う研究は存在するが（Generative Agents, CALM-DT等）、因果推論の文脈で体系的に活用する研究は未開拓。

## 2.3 統合的な課題認識

```
[課題1] 因果推論にはDAGの事前定義が必要 → 複雑環境では定式化困難
[課題2] LLMは環境シミュレーションに使える可能性がある → 因果推論との接続が未確立
[課題3] シミュレーションの出力が「知識想起」と区別できなければ価値がない

→ 研究問い: LLMマルチエージェントシミュレーションにより、
   事前のDAG定義なしに、知識想起では検出できない因果関係を発見できるか？
```

## 2.4 なぜ内部統制を題材にするか

| 理由 | 説明 |
|------|------|
| 実務的接点 | 研究者（Yuji）のドメイン専門性が直接活用できる |
| 複雑性の適切さ | 単純すぎず（DAGで十分）、複雑すぎない（検証可能）。人間の判断・相互作用・組織文化が絡む「ちょうどよい複雑さ」 |
| 検証可能性 | 実際の監査データ（ERPログ、指摘事項DB）と照合可能 |
| 汎用性の主張 | 方法論は一般的に定式化し、実証は内部統制で行う。「他ドメインへの適用可能性」はdiscussionで主張 |
