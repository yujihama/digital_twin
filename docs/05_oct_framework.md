# 05. OCTフレームワーク設計

## 5.1 コンセプト定義

> **Organizational Causal Twin (OCT):** 組織の業務環境（プロセス、人間、統制、リスク）をLLMマルチエージェントシミュレーションで再現するデジタルツインであり、このツインへの介入（do演算）を通じて、事前にDAGを定義することなく因果効果を推定するフレームワーク。

## 5.2 従来アプローチとの比較

| 観点 | 従来の因果推論 | OCT |
|------|-------------|-----|
| 因果推論の手順 | DAG定義 → 変数計測 → 統計的推定 | 環境構築 → シミュレーション → 介入 → 観察 |
| DAGの必要性 | 必須（事前に研究者が定義） | 不要（LLMが環境を暗黙的にモデル化） |
| 反事実の生成 | 統計的推定（PSM, DR等） | シミュレーションで直接生成 |
| 未観測交絡の扱い | 識別不能（仮定に依存） | LLMの世界知識が暗黙的に内包 |
| 創発的因果パスの検出 | 不可能（DAGに含まれないパスは検出不能） | 可能（エージェント相互作用から創発） |

## 5.4 構成要素

### [1] Context Layer（文脈層）
組織の構造、プロセス、規則、文化を自然言語で記述。実データ（マニュアル、議事録、規程）から構築。

### [2] Agent Layer（エージェント層）
組織内の役割をシミュレートするLLMエージェント群。各エージェントは以下を持つ。
- **ペルソナ:** 役割、性格特性、経験年数、目標
- **メモリ:** 過去の行動と結果の記録
- **LLM:** 行動決定器（結果予測器ではない）

### [3] Environment State（環境状態）
業務フロー、財務データ、統制パラメータ、時間等を管理する独立モジュール。**環境の状態遷移はルールベースで管理**（LLMが「想像」しない）。

### [4] Intervention Interface（介入インターフェース）
自然言語で介入を記述 → 環境パラメータの変更として実装。
- 例: 「購買承認の金額閾値を100万→500万に変更」
- 例: 「月次棚卸を四半期に変更」
- 例: 「経理部長が退職し、経験3年の後任が着任」

### [5] Observation Logger
全ステップの環境状態、エージェント行動、相互作用を構造化して記録。因果分析の入力データとなる。

## 5.5 設計原則：Mode R（知識想起）への退化を防ぐ制約

| 制約 | 実装 | 目的 |
|------|------|------|
| 1ステップ1行動 | 各LLM呼び出しは「次の1行動」のみを決定 | LLMが結論を「想像」することを防止 |
| 部分観測 | 各エージェントは役割に応じた情報のみ観察 | LLMの全体知識によるショートカット防止 |
| メモリの局所性 | 各エージェントは自分の経験のみ記憶 | 暗黙的通信の排除 |
| 環境の物理的制約 | 処理件数上限、営業時間等をハードコード | 制約無視の防止 |
| 外部ショック注入 | ランダムイベント（人事異動、取引先変動等） | 学習データにない状況の生成 |
| 行動順序ランダム化 | 各ステップでエージェント行動順をシャッフル | 暗黙的因果構造の排除 |

**最重要原則:**
```
LLMは環境状態を「想像」しない。
環境状態は独立モジュールがルールベースで管理する。
LLMの役割は「行動の選択」であり「結果の予測」ではない。
```

## 5.6 因果効果の推定方法

### 従来パラダイム（観察データからの統計的推定）
```
ATE = E[Y(1)] - E[Y(0)]
    = 観察データから傾向スコア等で推定
    → 識別条件（無交絡、重複）の成立が前提
```

### OCTパラダイム（シミュレーションベース）
```
ATE_sim = E[Y_twin(do(T=1))] - E[Y_twin(do(T=0))]
        = ツインで統制あり/なしをそれぞれN回シミュレーション
        → ツインの忠実度（fidelity）の検証が前提
```

DTCFの枠組み: factual検証 + sensitivity analysis + boundsの提示。
